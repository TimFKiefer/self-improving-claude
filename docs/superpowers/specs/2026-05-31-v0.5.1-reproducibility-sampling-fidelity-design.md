# v0.5.1 — Reproducibility Check + Sampling-Fidelity Fix — Design Spec

- **Date:** 2026-05-31
- **Status:** Approved in brainstorm; awaiting user review before `writing-plans`.
- **Milestone:** v0.5.1 patch per ROADMAP §v0.5.0 — closes the two items explicitly deferred to v0.5.1: (1) the reproducibility check (`>50%` keep-set overlap on rerun) and (2) acting on the load-bearing v0.5.0 lesson that *sampling fidelity matters as much as gate discipline*.
- **Scope:** Add a **confirmation re-run** stage to the auto-loop so a candidate keep must survive a best-of-3 fresh re-check before it is committed (kills noise-driven keeps inline). Add an **overlap-measurement** tool. Execute **one fresh layered run** from the pre-RC baseline and compare its keep-set to the v0.5.0 RC keep-set. Document the **variance budget** and codify the opus+max fidelity config.
- **Companion:** `docs/knowledge/eval-methodology.md` (variance budget lands here); v0.5.0 spec `2026-05-28-v0.5.0-auto-loop-design.md` §G6 (the reproducibility goal this pays off); the RC audit run `prompt-lab/auto-runs/2026-05-29T072722Z-sonnet/`.

## 1. Why this work exists

v0.5.0 shipped the auto-loop and tagged it. Its load-bearing lesson, recorded in the release commit and ROADMAP: **single-shot eval variance (~±3 points/fixture) is large enough to wash out the small score deltas the ratchet gates on.** The first RC run (haiku skill-runner, default effort, 50 iter) kept 5 commits that all passed their in-iter held-out gates; a *fresh* re-check — same code, different scores — showed the held-out had drifted, and all 5 were reverted. Only the opus + max-effort re-run produced keeps that survived a fresh re-check.

Two consequences were deferred to v0.5.1:

1. The gate trusts a **single** measurement. A noise spike that happens to clear epsilon on one eval gets committed. The fix is to make a keep *prove itself more than once* before it lands.
2. The spec's reproducibility criterion (§G6: re-running yields `>50%` keep-set overlap) was never measured — a rerun is its own expensive run.

These are coupled: **the reproducibility check is the fitness test for the sampling-fidelity fix.** A noisy loop produces low keep-set overlap by construction; a loop that only commits confirmed gains should reproduce its keeps more faithfully.

Per VISION.md: *"Git is the ratchet. It guarantees the system can only move forward: every kept change is a proven improvement."* v0.5.0 made the ratchet real; v0.5.1 makes "proven" mean *proven against noise*, not *lucky once*.

## 2. Goals

- **G1 — Confirmation re-run.** When an edit passes the existing visible `strictly_better` gate **and** the held-out `regresses` gate, it must additionally survive a **best-of-3 majority** re-check before commit: the visible gain holds in ≥2 of 3 target measurements **and** held-out never regresses across all 3 measurements. Otherwise the edit is reset (`rejected: confirmation_failed`).
- **G2 — Reproducibility measured.** Execute one fresh layered run from the pre-RC baseline (`15cb51e`) under the RC config + confirmation re-run, and report keep-set overlap against the RC keep-set (`c6c5bac`, `f627357`, `4915a20`). The number is reported either way — `>50%` is the spec's success bar, but `<50%` is a valid, documented finding that informs v0.6, **not** a release blocker.
- **G3 — Variance budget documented + fidelity config codified.** A "Variance budget" section in `docs/knowledge/eval-methodology.md` captures the ±3-point single-shot variance, why opus+max + confirmation re-runs are the response, and the reproducibility caveats. The canonical fidelity command is documented in `evals/results/README.md`.
- **G4 — Confirmation measurements audited.** Every confirmation re-check (all target + held-out measurements + verdict) is recorded on the iteration's audit row, so the log shows *why* a candidate was confirmed or rejected.

## 3. Non-goals

- **No full N-sampling of every eval.** Only *candidate keeps* (edits that already cleared both in-iter gates) get re-run. Re-measuring all 9+3 fixtures every iteration would multiply cost ~3× for no benefit on the >90% of iterations that reject at the first gate.
- **No proposer-determinism engineering.** Reproducibility also depends on proposer stochasticity and rotation-order sensitivity, not just eval noise. We **document** this as a caveat; we do not change proposer temperature, fix a rotation seed, or otherwise force determinism in v0.5.1.
- **No two-run gold-standard reproducibility.** Two fresh layered runs (~$230) would be the cleanest number. v0.5.1 compares one fresh run against the existing RC keep-set (~$140) and treats RC as the reference. The methodology-mismatch (RC had no confirmation stage) is a stated caveat.
- **No new grader axes; no frontmatter/trigger optimization.** Those remain v0.6 (activation frontier). v0.5.1 touches the loop's *keep decision*, not what it measures or which file it edits.

## 4. Architecture

### 4.1 Confirmation re-run inside `run_iteration`

The existing `run_iteration` (`evals/auto_loop.py`) already produces, for a candidate edit:
- `t1` — the post-edit **target** measurement (step 4: `_run_eval_and_extract_result`)
- `h1` — the post-edit **held-out** measurement (step 6: `run_holdout_eval`)

The confirmation stage inserts between the existing held-out check and `git_commit_iteration`. Early-exit ordering is preserved so cost is only spent on edits that already look like keeps:

```
# step 4: t1 = eval(target)
if not strictly_better(t1, baseline):        reset; reject "no_visible_gain"     # early exit, no confirm
# step 6: h1 = eval(holdout)
if regresses(h1, holdout_baseline):          reset; reject "holdout_regression"  # early exit, no confirm

# NEW step 6.5: confirmation re-run (best-of-3, k=2 extra measurements)
targets  = [t1]; holdouts = [h1]
for _ in range(confirmation_reruns):         # default 2
    targets.append(eval(target))
    holdouts.append(eval(holdout))
visible_ok  = sum(strictly_better(t, baseline) for t in targets) >= majority(len(targets))   # ≥2 of 3
holdout_ok  = all(not regresses(h, holdout_baseline) for h in holdouts)                       # all 3
if not (visible_ok and holdout_ok):          reset; reject "confirmation_failed"
commit; keep                                  # baseline/holdout advance to the *last* confirmed measurement
```

- `confirmation_reruns` is a parameter (default **2** → best-of-3), threaded from a new CLI flag `--confirm-reruns`. Setting it to `0` reproduces the v0.5.0 behavior exactly (used by cheap dev runs and as an α-compat path).
- `majority(n) = n // 2 + 1` → for n=3, requires 2.
- On a kept edit, `state.holdout_baseline` advances to the **last** confirmation measurement (consistent with the current "update to post-edit holdout" logic), and the visible/target baseline is invalidated so the next pick re-evals — unchanged from today.

### 4.2 The reproducibility run (operational, not a code artifact)

1. Branch `repro/v0.5.1` off current `main` (carries the new harness, incl. confirmation logic).
2. Reset **only the slow-state** to the pre-RC baseline: `git checkout 15cb51e -- plugin/skills/_shared plugin/skills/improve plugin/skills/improve-init`, then commit on the branch. This yields **new harness + pre-RC skill body** — the exact starting point the RC run discovered its keeps from. `main` and the `v0.5.0` tag are never touched.
3. Run with the RC config plus confirmation:
   `python3 -m evals.auto_loop --proposer claude-sonnet-4-5 --skill-runner opus --judge opus --effort max --rotate-bottom-n 3 --max-iterations 20 --confirm-reruns 2 --max-usd 200 --max-hours 10`
4. Cost/time: **~$140, ~9–10h** (the ~$114 RC envelope + ~$30 for confirmation re-runs on the handful of candidate keeps). One run.
5. After it finishes, run the overlap tool (§4.3) against the RC audit dir; record the verdict; delete or archive the branch (no merge to `main`).

### 4.3 Overlap measurement — `evals/reproducibility.py`

Pure module + thin CLI. Loads two run dirs' `iterations.jsonl`, extracts kept rows (`decision == "kept"`, with `edit`/`hypothesis`/`fixture`/`commit_sha`), and reports overlap two ways:

- **Headline (advisory, judge):** for each RC keep, ask one judge call "did the new run produce a substantively equivalent improvement? (yes/no + one-clause reasoning)". Overlap = yes-count / |RC keeps|. Mirrors the project's "deterministic gate + advisory judge, human decides" pattern — the judge informs, the human makes the final `>50%` call.
- **Floor (deterministic):** by-fixture keep-set overlap = |fixtures kept by both| / |fixtures RC kept|. No model call; a lower bound that's robust to paraphrase under-counting.
- **Report:** prints both numbers plus a side-by-side table of RC keeps vs new keeps (fixture, hypothesis, edit anchor) so the decision is auditable by eye.

The RC keep-set reference is read from the RC audit dir (`prompt-lab/auto-runs/2026-05-29T072722Z-sonnet/iterations.jsonl`), not hardcoded — so the tool generalizes to future repro checks.

## 6. Testing

- **Unit only in the suite — no real model calls.** Confirmation tests stub the eval functions to return scripted score dicts (e.g. `[gain, noise, gain]` → kept; `[gain, noise, noise]` → rejected; held-out `[ok, ok, regress]` → rejected). Reuse the existing `test_auto_loop.py` fakes/patching style.
- **Overlap tests** load small hand-authored `iterations.jsonl` fixtures and assert the by-fixture floor and the keep-extraction; the judge path is tested with a fake client returning canned verdicts.
- Target test count: ≥ existing + ~8 new. Full suite green before the run.
- The reproducibility **run** is validated operationally (audit log shows confirmation rows; cost/time within caps), not in the unit suite.

## 7. Risks & open questions

- **R1 — Reproducibility may come back `<50%` even with confirmation.** Likely if proposer stochasticity + rotation sensitivity dominate over eval noise. Mitigation: this is a *finding*, not a failure (§G2). The side-by-side report makes the cause legible; if path-dependence dominates, that's explicit v0.6 input.
- **R2 — Methodology mismatch (RC had no confirmation stage).** The comparison is "do RC's keeps re-emerge under the stronger loop," not pure run-to-run repro. Documented as a caveat in the result note; the two-run gold standard remains available later.
- **R3 — Cost overrun.** `--max-usd 200` caps it; the coarse estimator over-counts, so real spend should land under. The confirmation re-runs add bounded cost (only candidate keeps trigger them).
- **R4 — Confirmation never triggers** (if the run produces no candidate keeps at all). Then the run validates the *gate*, not the confirmation logic — but the unit tests already cover the confirmation branch deterministically, so coverage doesn't depend on the run.
