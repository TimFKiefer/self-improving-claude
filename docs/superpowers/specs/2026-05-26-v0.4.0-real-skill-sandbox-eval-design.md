# v0.4.0 SP-1 — Real-Skill Sandbox Eval + Restraint Fixtures — Design Spec

- **Date:** 2026-05-26
- **Status:** Approved in brainstorm; awaiting user review of this written spec before `writing-plans`.
- **Milestone:** First sub-project (SP-1) of **v0.4.0 "Foundation for Autonomy"** (see `docs/ROADMAP.md`).
- **Scope:** the `evals/` measurement layer — a new headless sandbox runner that drives the *real* `/improve` / `/improve-init` slash commands end-to-end, restraint fixtures, and a Claude-only re-baseline. No changes to the orchestrator skill bodies, the graders, or the legacy prompt path.
- **Companion backlog:** `docs/superpowers/plans/2026-05-24-self-improving-claude-v0.4-backlog.md` (Track 6.2 + 6.4).

---

## 1. Why this work exists

The conformance eval is the instrument that gates the project's autonomous change→test→score→ratchet loop (`docs/VISION.md`; `docs/knowledge/eval-methodology.md` §1). Today it measures a **proxy**, not the product:

- `evals/run.py:64` assembles its prompt from `evals/prompt_template.md` (a 69-line hand-written form-selection prompt) plus the five `improve/references/*.md`. **It never reads `SKILL.md`.** The orchestrator's actual procedure — Step 1 request-classification, Step 2 inspect-existing-config, Step 6 self-critique, the Step 7 stderr-discipline retry loop, the Step 8 approval loop, Step 9 file writes — is **untested**. A `SKILL.md` regression passes green.
- The form ladder now lives in **three** places that must stay in sync (`SKILL.md` Step 4, `prompt-rubric.md`, `prompt_template.md`).
- Every fixture is a positive case ("propose X"). **Restraint** — the trust-critical "propose nothing" dimension — is unmeasured. A loop optimizing against a positive-only suite is rewarded for over-proposing.

This sub-project closes the proxy gap by running the **literal slash command** in a throwaway project and grading what the real skill produces, and adds restraint fixtures so "correctly proposing nothing" becomes a score the loop can optimize toward.

### v0.4.0 exit criteria addressed

| Exit criterion (ROADMAP) | This spec |
|---|---|
| `evals/run.py` drives the actual `SKILL.md` (not a separate template) | ✅ — the sandbox runner invokes the real `/improve` command; `run.py` gains the sandbox mode as the canonical path |
| At least 2 negative / restraint fixtures present | ✅ — fixtures 011 + 012 with `expect_no_proposal` |
| All prior baselines re-run against the new eval; deltas committed | ✅ **(amended)** — see §1.1 |

**Out of this sub-project** (other v0.4.0 exit criteria, handled by later SP's): the sandbox **firing** harness beyond exists-and-parses (full 6.1), references single-sourcing + `@../` re-probe (7.2/7.3), knowledge re-grounding (7.1). Track 6.3 (noise-penalizing aggregation + fixed grader) **already shipped in v0.3.4** and is not re-scoped.

### 1.1 Amendment to exit criterion #5 (decided in brainstorm)

The real skill can only be exercised on a model that has a Claude Code runtime (skills, tools, slash commands). **gemma-via-Ollama and the raw `anthropic` SDK cannot execute a slash command**, so the sandbox eval is inherently **Claude-models-only**.

**Decision:** the sandbox real-skill eval becomes THE canonical conformance eval for v0.4.0; **gemma is dropped** from the canonical baseline. Exit criterion #5 is reworded to *"all CC-capable backends (Haiku / Sonnet / Opus) re-run against the new sandbox eval; deltas committed."* The cost is losing the free, no-subscription local baseline — accepted deliberately. The legacy prompt path in `run.py` is **retained, not deleted** (it is still imported by `evals/benchmark.py`), but is no longer the canonical conformance instrument.

## 2. Goals

- **G1.** The eval exercises the real orchestrator: the literal `/improve` (reactive) / `/improve-init` (proactive) command runs end-to-end in a sandbox — real plugin load, real Steps 1–9, real Read/Grep/Bash tool use, real file writes.
- **G2.** The orchestrator skill bodies stay **byte-for-byte unchanged**. Non-interactivity is imposed at *invocation* time, not by editing the measured artifact.
- **G3.** Proposal quality is graded by the **existing** `grade_code` / `grade_model`, unchanged — fed the skill's emitted JSON summary of surviving candidates (which also captures `claude-md-note`s that Step 9 never writes to disk).
- **G4.** A separate, orthogonal **integrity** axis confirms the skill actually wrote valid config (settings.json + hook files exist and parse) — the seed of the future 6.1 firing harness, reported as `install_rate`, never blended into the quality grades.
- **G5.** Restraint is measurable: a fixture can assert "the correct outcome is zero proposals," scored binary (10/0), and excluded from the quality averages.
- **G6.** The Claude baseline (Haiku / Sonnet / Opus) is re-run against the new eval and committed, with the Opus leg quota-gated behind explicit user approval.
- **G7.** Everything except the gated live runs is unit-tested with **no** `claude` invocation, no network, no quota (mocked `subprocess` + a real `tmp_path`).

## 3. Non-goals

- **Full behavioral firing verification (6.1).** We stop at "the written hook exists and parses." Installing it into a live session and asserting it *fires* (exit-2 blocks, stderr emitted, formatter ran) is a later sub-project.
- **Track 7** (single-sourcing references, `@../` re-probe, knowledge re-grounding, the orchestrator dual-copy).
- **Editing `SKILL.md`** (see G2). The fallback in §6.3 is the only scenario that would, and only if the spike forces it.
- **Deleting the legacy prompt path / Ollama client / gemma fixtures.** Retained for `benchmark.py`.
- **Changing `grade_code` / `grade_model` / `assemble_prompt` signatures or behavior.** `benchmark.py` imports `assemble_prompt`, `parse_proposals` from `run.py` and must keep working untouched.
- **Multi-sample (N>1).** This is a regression tripwire, not a leaderboard. N=1 per fixture×model; non-determinism is flagged, not averaged away.

## 4. Architecture

```
fixture (dataset.json + fixtures/<id>/)
        │
        ▼
evals/sandbox_runner.py        ← NEW: owns the headless invocation
  build temp project ──► invoke `claude --print …` ──► capture JSON echo + read disk ──► teardown
        │
        ▼
evals/run.py (sandbox mode)    ← MODIFIED: dispatch reactive→/improve, proactive→/improve-init,
  parse echo → proposal dicts → grade_code / grade_model (UNCHANGED)
  + disk integrity → install_rate
  + restraint scoring for expect_no_proposal fixtures
        │
        ▼
evals/results/2026-05-26-v0.4.0-sandbox-<model>.json
```

### 4.1 `evals/sandbox_runner.py` (new)

Single responsibility: run one fixture against one model through the real CLI and return structured evidence. Pure orchestration of `subprocess` + filesystem; no grading.

```python
def run_in_sandbox(*, entry: dict, fixture: Fixture, model: str,
                   plugin_path: Path, timeout: float = 600) -> dict:
    """Drive the real slash command headlessly for one (fixture × model).

    Returns:
      {
        "echo": [ <proposal dict>, ... ],   # parsed from the skill's JSON summary; [] if none/parse-fail
        "echo_valid": bool,                  # the final JSON summary parsed cleanly
        "written": {                         # what actually landed on disk in the sandbox
            "settings_parses": bool,
            "hook_files": ["<slug>.py", ...],
            "permission_rules": ["Read(**/.env*)", ...],
        },
        "raw_result": "<final assistant text>",
        "returncode": int,
        "error": str | None,
      }
    """
```

Behavior:
1. **Build the sandbox.** `tmp = mkdtemp()`. Write `fixture.project_files` as real files under `tmp/`. If `fixture.telemetry`, write it to `tmp/.claude/self-improving-claude/telemetry.jsonl`. Ensure `tmp/.claude/` exists.
2. **Choose the command** by `entry["trigger"]`: `improve` → `/improve`, `improve-init` → `/improve-init`.
3. **Compose the invocation** (exact flags pinned in the §6.1 spike):
   - `claude --print --model <cli_model> --plugin-dir <plugin_path> --permission-mode acceptEdits --allowedTools "Read,Write,Edit,Bash,Grep,Glob" --output-format json`
   - positional prompt: `"/improve <user_args>"` (or `/improve-init`).
   - `--append-system-prompt "<OVERRIDE>"` (the §4.4 text). For **reactive** fixtures, the override also carries the labeled recent-conversation block built from `fixture.chat` (a one-shot headless run has no prior conversation to supply `<recent_chat>`).
   - `cli_model` is mapped by reusing `evals.client_claude_cli._to_cli_model` (so Sonnet stays the verbatim `claude-sonnet-4-5`, not the credit-gated 1M `sonnet` alias).
4. **Run** via `subprocess.run(..., cwd=tmp, capture_output=True, text=True, timeout=timeout)`.
5. **Capture the echo.** Parse `--output-format json` → `.result` (final assistant text) → extract the trailing JSON object (reuse `run.parse_proposals` semantics: tolerate optional fences). Empty/parse-fail → `echo=[]`, `echo_valid=False`.
6. **Read disk.** Parse `tmp/.claude/settings.json` (record `settings_parses`); list `tmp/.claude/hooks/*`; collect `permissions.deny`/`permissions.ask` rule strings.
7. **Teardown** `tmp` (always, even on error).

The runner makes **exactly one** `claude` invocation per call. No retries, no incremental-save wrapper (per the `simple-reruns-over-resilience-scaffolding` feedback memory — if a run dies on a usage limit, the operator re-runs the plain command).

### 4.2 `evals/run.py` (modified)

- New **sandbox mode**, selected explicitly (`--sandbox` flag or `EVAL_MODE=sandbox`), running **one model per invocation** (model via `SANDBOX_MODEL` env, default `haiku`), mirroring how the v0.3.4 per-backend baselines were produced. Output: `evals/results/<date>-v0.4.0-sandbox-<model>.json`.
- For each entry: call `run_in_sandbox`; take `result["echo"]` as the proposal list; grade with the **existing** `grade_code(p, entry["expected_hook_traits"])` and `grade_model(proposal=p, planted_problem=entry["planted_problem"], client=<Haiku grader>)`. The grader stays pinned to Haiku via `grade_model.GRADER_MODEL` (a normal `ClaudeCliClient`, prompt path) — only the *proposer* changes to the sandbox run.
- **Integrity:** per proposal, compute `installed_ok` (§4.5) from `result["written"]`; aggregate `install_rate` per fixture and overall.
- **Restraint:** entries with `expect_no_proposal: true` take the §5 branch and are excluded from `average_code` / `average_model`.
- The **legacy prompt path** (`assemble_prompt` → single `client.messages.create`) remains the default mode and is unchanged. `assemble_prompt` / `parse_proposals` signatures are untouched, so `benchmark.py` is unaffected.

### 4.3 The JSON echo = the existing proposal shape

The override instructs the skill, after really running Steps 1–9, to emit a single JSON object of **every surviving candidate** in the exact shape `grade_code` already documents (`evals/grade_code.py:3`):

```json
{"proposals": [
  {"form": "...", "event": "...", "matcher": "...", "rationale": "...",
   "script_lang": "...", "script": "...full body...", "prompt": "...",
   "rule": "...", "claude_md_line": "...", "sentinel_name": "self-improving-claude/<slug>"}
]}
```

This is the *same artifact the proxy used to emit* — but now it is the **output of the real skill run** (real procedure, real tool use, real writes), not a one-shot proxy prompt. Grading is therefore identical to today; only the fidelity of how proposals were produced improves. Full `script` bodies must be echoed so `script_parses` / `imperative_stderr` keep working.

### 4.4 The non-interactive override (verbatim, passed via `--append-system-prompt`)

> You are running NON-INTERACTIVELY inside an automated evaluation. The `AskUserQuestion` tool is unavailable — wherever the procedure tells you to ask the user, instead choose the single best option yourself using the rubric and continue. Do not wait for input. Auto-apply every surviving candidate as if the user approved it (perform Step 9 writes). Skip the Step 10 rewind/restart prose. When the procedure is complete, output as your final message a single JSON object and nothing else: `{"proposals":[ … ]}`, one entry per surviving candidate, each using the exact field shape from the orchestrator's output contract (form, event, matcher, rationale, script_lang, script, prompt, rule, claude_md_line, sentinel_name). Include candidates of every form, including `claude-md-note` candidates even though you do not write those to disk.

For reactive fixtures, append a labeled block: *"Recent conversation the user is reacting to (treat as `<recent_chat>`):\n\n<fixture.chat>"*.

`AskUserQuestion` is additionally omitted from `--allowedTools` as a second line of defense.

### 4.5 Integrity check (`installed_ok`, → `install_rate`)

Per echoed proposal, confirm it actually landed (only meaningful when `settings_parses`):

| Form | `installed_ok` is true when |
|---|---|
| `command-hook` | a hook file matching the sentinel slug exists under `.claude/hooks/` AND a matching hook entry is present in settings.json |
| `permissions.deny` / `permissions.ask` | the `rule` string appears in the corresponding settings.json permissions array |
| `claude-md-note` | N/A — Step 9 never writes these (skill outputs the line for the user to paste); excluded from `install_rate` |
| `prompt-hook` | a matching hook entry of `"type":"prompt"` is present in settings.json |

`install_rate` = installed_ok / (proposals where it applies), per fixture and overall. Reported alongside `clean_rate`; **never folded into `code`/`model` grades** (mirrors v0.3.4's separate-axes philosophy).

## 5. Restraint fixtures (6.4)

### 5.1 Schema + scoring

`dataset.json` entries gain an optional `"expect_no_proposal": true`. Such entries:
- omit `expected_hook_traits`;
- score **restraint = 10** iff `echo == []` **and** the run wrote no hook files and added no permission rules; else **0**;
- are excluded from `average_code` / `average_model`; contribute to a new `average_restraint`.

`run.py:_aggregate` gains the branch; the summary schema adds `average_restraint` and per-entry `{"restraint": 10|0}` for these fixtures. Binary and deterministic — directly loop-gateable per `VISION.md`.

### 5.2 The two fixtures

- **`011-no-overblock-deny`** (proactive, `trigger: improve-init`). A healthy project whose telemetry shows normal, successful tool use and whose layout contains tempting-but-legitimate targets (e.g. `.env.example` checked into the repo, frequent `Bash(rm …)` in a build script). There is **no observed harmful behavior**. Correct outcome: propose nothing — a speculative `Read(**/.env*)` deny or `Bash(rm:*)` block would impede legitimate work and isn't justified by observed evidence. Tests Step 3's "observable, not hypothetical" discipline.
- **`012-one-off-bug-no-guardrail`** (reactive, `trigger: improve`). `fixture.chat` shows Claude making a **one-off** mistake the user already corrected (e.g. referenced the wrong local variable once), with no generalizable, deterministically-detectable pattern. Correct outcome: nothing warrants a standing guardrail. Tests restraint against the reflex to always emit a hook.

Both are authored so the genuinely-correct answer is zero proposals; if the real skill over-proposes, restraint scores 0 and surfaces the gap.

## 6. Execution details

### 6.1 Feasibility spike — gating Task 0

Before building the runner, confirm against the **installed** Claude Code version (the guide-agent's version numbers were stale; treat all flags below as candidates to pin, not facts):
1. `claude --print "/improve '…'" --plugin-dir <repo>/plugin …` actually **executes** the plugin's slash command headlessly (vs. treating it as literal text). Note: `client_claude_cli.py` already proves `claude --print --model …` works; it currently passes `--disable-slash-commands`, which the sandbox path must NOT.
2. Plugin discovery from an arbitrary `cwd` (which flag: `--plugin-dir` or other).
3. `AskUserQuestion` behavior under `--print` (error vs. skip) — validates whether the §4.4 override alone suffices or the §6.3 fallback is needed.
4. Auto-approve writes (`--permission-mode acceptEdits` vs. alternatives) so Step 9 lands files.
5. `--output-format json` shape and where the final assistant text lives.
6. `--append-system-prompt` coexists with a slash-command invocation, and can carry the reactive-chat block.

Deliverable: a pinned, working `claude` invocation (one fixture, one model) + a short note of the confirmed flags. This is a live run → consumes subscription quota; run it on Haiku.

**Spike result (2026-05-26 — ran, pins the design):** ✅ feasible. Claude Code 2.1.150 exposes all needed flags. `/improve-init` executes headlessly via `--plugin-dir`; `--output-format json` → `.result`. Two corrections to the as-written design, now pinned:
1. **The runtime override must be FORCEFUL.** The mild "AskUserQuestion is unavailable, choose the best option" phrasing let the skill's Step 8 approval dominate (approval prose, no JSON, no writes). The forceful "ABSOLUTE DIRECTIVE … never ask … perform writes now … final message is JSON only" phrasing works. (G2 — pristine `SKILL.md` — still holds; the override is runtime via `--append-system-prompt`.)
2. **Permission mode must be `bypassPermissions`, not `acceptEdits`.** `acceptEdits` denied the Step-9 `Write`/`Bash`; `bypassPermissions` (user-authorized 2026-05-26; safe in a throwaway temp dir) lands the writes with 0 denials. No `--allowedTools` needed under bypass. **Do not use `--bare`** (it disables OAuth).

**Added risk (validity):** the forceful override can push smaller models off-rubric (one Haiku run proposed a command-hook instead of the gold `permissions.deny`; another chose correctly). At N=1 this is variance — read gross drift, not fine deltas; the baseline report must caveat that proposals are produced under a coercive non-interactive override. (A future cleaner alternative — a first-class non-interactive mode in the orchestrator — was considered and deferred; chosen path keeps `SKILL.md` pristine + real writes + the `install_rate` integrity axis.)

### 6.2 Baseline rerun (Task: gated, operational)

After the runner + fixtures + tests are green: run the full dataset (positive + restraint) on **Haiku, then Sonnet**, commit `evals/results/2026-05-26-v0.4.0-sandbox-{haiku,sonnet}.json`, then **stop for explicit user go-ahead before the Opus leg** (premium quota; matches the v0.3.4 gate and the `simple-reruns` memory). N=1 per fixture×model. Output is non-deterministic — do not assert exact scores; report the leaderboard delta vs. the v0.3.4 prompt-path baseline with the caveat that the *instrument changed* (real skill vs. proxy), so it is a re-baseline, not a trend line.

**Cost note.** Each cell is now a full agentic procedure (multiple tool calls), not a single completion — materially more expensive than the v0.3.4 prompt path. Deliberate.

**Thinking/effort config.** The `benchmark-thinking-config` memory (extended thinking + max effort + Opus judge) governs the **capability benchmark** (`benchmark.py`), *not* this Haiku-graded conformance run. This baseline runs on the plain subscription path unless the user directs otherwise.

### 6.3 Fallback (only if the §6.1 spike fails)

If `claude --print` cannot execute the plugin slash command headlessly, fall back to feeding the **resolved skill body** (the real `SKILL.md` with its `@references/*.md` inlined) as the sandbox prompt with the same override and tools enabled — still real tool use + real disk writes + echo grading, just not the literal `/` entry point. This preserves G1's substance (real procedure, real writes) while losing only the command-dispatch/frontmatter fidelity. Editing `SKILL.md` to add an explicit non-interactive mode is the last resort and would be flagged for a separate decision (it touches the dual-copy that Track 7 owns).

## 7. Testing

Unit tests make **no** `claude` call (mock `subprocess.run`) and use a real `tmp_path`:

- `evals/tests/test_sandbox_runner.py` (new):
  - builds `fixture.project_files` + telemetry into the sandbox correctly;
  - constructs the expected `argv` (model mapping via `_to_cli_model`, `--plugin-dir`, `--permission-mode`, `--allowedTools` without `AskUserQuestion`, correct command per trigger, override present; reactive carries the chat block, proactive does not);
  - parses the echo from a canned `--output-format json` payload; empty/garbage → `echo=[]`, `echo_valid=False`;
  - integrity: pre-seed fake `.claude/settings.json` + `hooks/<slug>.py` → `installed_ok` true; missing → false; unparseable settings → `settings_parses` false;
  - teardown removes the temp dir even when the subprocess "fails".
- `evals/tests/test_run.py` (extend): sandbox-mode aggregation — `install_rate` computed and kept separate from grades; restraint branch scores 10 on empty-echo-and-no-writes, 0 otherwise, and is excluded from `average_code`/`average_model`; legacy prompt mode still works unchanged.
- The §6.1 spike and §6.2 baseline are live/integration steps — marked and skipped in the default `pytest -m "not integration"` sweep.

## 8. Risks & mitigations

| Risk | Mitigation |
|---|---|
| **Feasibility linchpin** — headless slash-command exec may not work as imagined | Task 0 spike gates everything; §6.3 fallback preserves the substance |
| **Non-determinism** of agentic runs at N=1 | Accept for a tripwire; flag in results; the regression signal is gross drift, not fine deltas |
| **Cost** (full procedure × fixtures × 3 models) | Opus-gated; Haiku+Sonnet first; one run per cell, no retries |
| **Echo vs. disk divergence** (skill claims X, writes Y) | `install_rate` integrity axis surfaces it; settings.json must parse |
| **Stale flag intel** from the guide agent | All flags pinned empirically in the spike before building |
| **Reactive-chat injection** placement (system-append vs. prompt body) | Spike confirms; system-append is the plan, prompt-body the fallback |
| **gemma drops out of CI** (no free local baseline) | Accepted (brainstorm decision); legacy prompt path retained for `benchmark.py` and any future no-key reference |
