# v0.5.0 — The Self-Improvement Loop (Path A) — Design Spec

- **Date:** 2026-05-28
- **Status:** Approved in brainstorm; awaiting user review before `writing-plans`.
- **Milestone:** v0.5.0 "The Self-Improvement Loop" — vision-aligned auto-loop per ROADMAP §v0.5 Path A.
- **Scope:** an end-to-end automated optimization loop that proposes one bounded edit to `plugin/skills/_shared/orchestrator-procedure.md` (or rubric / examples) per iteration, runs the v0.4 sandbox eval against the visible 9 fixtures, confirms against the held-out 3, and ratchets via git (commit if better, reset if worse). Runs unattended, capped on cost / time / iterations, with full audit log. Optimizes the skill *body* only — trigger frontmatter is v0.6.
- **Companion:** `docs/knowledge/prompt-engineering.md` §8 (SkillOpt disciplines); `docs/knowledge/agentic-patterns.md` §2d (evaluator-optimizer); `docs/knowledge/eval-methodology.md` (the 5-step eval workflow this automates).

---

## 1. Why this work exists

The project's name is a promise. v0.1–v0.4 built the substrate: a real-skill sandbox eval, a held-out validation subset, slow/fast state separation enforced by `sync_skills.py`, compactness tracking, and a knowledge base that codifies the optimization discipline (SkillOpt §8). The v0.4 prompt-lab loop ran the auto-improve cycle *by hand* — five candidates, one kept (C1′), two rejected (C2, C3), one folded back (Phase 1+2) — and produced measurable gains across all three models. **v0.5 codifies that cycle as an unattended process.**

Per VISION.md: *"The name of this project is a promise. 'Self-improving' is not a metaphor. It is a loop we intend to actually run."* v0.5 is where that promise either gets paid or gets quietly converted into a different project.

---

## 2. Goals

- **G1.** A single command — `python3 -m evals.auto_loop` — runs N iterations unattended and writes an audit log.
- **G2.** The loop optimizes against the same eval the human prompt-lab loop used (`evals/run.py --sandbox`). No new grader; no new metrics. Verifier reuse is structural.
- **G3.** Held-out gating is enforced before any commit. An edit that improves visible-9 but regresses held-out-3 is reverted.
- **G4.** Every iteration is recorded — proposer prompt, proposer response, applied edit, scores before/after on both subsets, keep/reset decision, hypothesis. Reading the log answers "what changed and why?"
- **G5.** Resource caps (USD, wall-clock, iteration count) are honored. The loop terminates cleanly on cap hit, no orphaned commits or partial state.
- **G6.** Reproducibility: re-running from the same baseline + same config produces directionally consistent improvements (keep-set overlap > 50%) even though individual decisions may differ on model nondeterminism.

## 3. Non-goals

- **No new grader axes.** The eval is the verifier. v0.6 may add per-Step binary assertions if the loop plateaus — not now.
- **No skill-frontmatter optimization.** Description / trigger accuracy is the v0.6 "activation frontier" theme. v0.5 optimizes the body only.
- **No external-skill generalization.** v0.8. v0.5 hardcodes paths to *this* repo's `_shared/` and `evals/`.
- **No unattended-overnight resume-from-crash.** v0.7. v0.5 stops cleanly on cap; resume is a stretch goal.
- **No composed PostToolUse + Stop hooks.** Path B; the loop may discover this form independently (the procedure could grow to surface it). If we hit v0.5.0 without it appearing, we re-evaluate in v0.5.x or v0.6.
- **No "bootstrapped binary assertions per Step heading."** The original v0.5 ROADMAP listed this; we explicitly defer. The eval already grades fixture-level; adding step-level checks is its own mini-project not required for MVP. Move to v0.6 only if the loop is starved for signal.

## 4. Architecture

### 4.1 The loop

```
state.visible_baseline = run_eval(--no-holdout)          # 9 fixtures
state.holdout_baseline = run_eval(--holdout-only)        # 3 fixtures (one-shot at start)
audit = open_audit_log()

for i in 1..max_iter:
    if caps_exceeded(): break

    fixture = pick_target(state.visible_baseline)         # rotate over bottom-3
    edit = edit_proposer(                                  # SDK / claude --print
        fixture=fixture,
        procedure=read("_shared/orchestrator-procedure.md"),
        rubric=read("_shared/references/prompt-rubric.md"),
        history=audit.last_n_rejected_edits(5),           # avoid retry
    )
    apply_edit(edit)
    run("scripts/sync_skills.py")                          # regenerate

    new_visible = run_eval(--no-holdout)
    if strictly_better(new_visible, state.visible_baseline, gating_metrics):
        new_holdout = run_eval(--holdout-only)
        if not regresses(new_holdout, state.holdout_baseline, gating_metrics):
            git.commit(message=auto_commit_message(edit, new_visible, new_holdout))
            state.visible_baseline = new_visible
            state.holdout_baseline = new_holdout
            decision = "kept"
        else:
            git.reset("--hard", "HEAD")
            decision = "rejected (held-out regression)"
    else:
        git.reset("--hard", "HEAD")
        decision = "rejected (no visible gain)"

    audit.write_iteration(i, fixture, edit, new_visible, decision, ...)

audit.write_summary()
```

### 4.2 Components

**`evals/auto_loop.py` — the driver.** New module. CLI via `python3 -m evals.auto_loop`. Lives next to `run.py` so it can import the existing eval machinery directly (no duplication of fixture loading, sandbox running, or aggregation).

**`evals/edit_proposer.py` — the proposer.** Thin wrapper around `claude --print` (Claude CLI, reuses `evals/client_claude_cli.py`). Prompt structure:

```
<role>
You are a prompt-optimization agent. Your task is to propose ONE bounded
edit to a Claude Code skill that will improve its eval score on a specific
failing fixture.
</role>

<discipline>
- One edit only. Add, delete, or replace — at most 8 lines total.
- Edit only these files: _shared/orchestrator-procedure.md, _shared/references/prompt-rubric.md, _shared/references/examples.md.
- Do NOT rewrite from scratch. Do NOT make multiple edits "in one go."
- Compactness matters — a deletion that doesn't regress the score is a win.
</discipline>

<failing_fixture>
{lowest-scoring fixture: id, planted_problem, expected_traits, actual proposal that scored low, code/model/install/fire scores}
</failing_fixture>

<current_procedure>
{full orchestrator-procedure.md}
</current_procedure>

<rubric>
{full prompt-rubric.md}
</rubric>

<recently_rejected_edits>
{last 5 edits the loop tried that didn't work, with why they were rejected}
</recently_rejected_edits>

<output_format>
Return JSON:
{
  "file": "_shared/orchestrator-procedure.md",
  "operation": "add" | "delete" | "replace",
  "anchor": "<unique string from the file that locates the edit>",
  "anchor_position": "before" | "after" | "replace",
  "new_content": "<the new text, or empty for delete>",
  "hypothesis": "<one sentence: why this edit should improve the failing fixture>",
  "confidence": 1-10
}
</output_format>
```

The anchor-based edit format means edits are applied via `Edit`-style substring matching, not line numbers (line numbers drift across iterations). The `anchor` field uniquely identifies the edit site; if the anchor isn't unique or doesn't exist, the edit is rejected without scoring.

**`evals/ratchet.py` — keep/reject logic.** Two functions:

- `strictly_better(new, old, gating_metrics)`: returns True iff at least one gating metric strictly improved AND none regressed beyond a small epsilon. Gating metrics: `average_code`, `install_rate`, `fire_rate`, `average_restraint`. Epsilon per metric (calibrated from baseline noise): `code 0.05`, `install 0.0`, `fire 0.05`, `restraint 0.0`.
- `regresses(new, old, gating_metrics)`: returns True if any gating metric dropped beyond epsilon. Used for held-out gate.

**`evals/audit.py` — the log.** Audit dir at `prompt-lab/auto-runs/<timestamp>-<proposer>/` contains:

- `config.json` — caps, models, seed.
- `iterations.jsonl` — one row per iteration with the iteration record (see below).
- `iterations/<i>/proposer-prompt.txt`, `proposer-response.json`, `eval-visible.json`, `eval-holdout.json` — full artifacts per iteration.
- `summary.md` — written on clean exit. Headline: "ran N iterations, kept K, scores X→Y on each metric, total USD spent."

Per-iteration record (one JSONL line):

```json
{
  "i": 7,
  "ts": "2026-05-28T20:14:00Z",
  "fixture": "010-block-staging-fetch",
  "edit": { "file": "...", "operation": "...", "anchor": "...", "new_content": "..." },
  "hypothesis": "...",
  "scores_visible_before": { "code": 7.4, "install": 0.80, ... },
  "scores_visible_after":  { "code": 7.6, "install": 0.85, ... },
  "scores_holdout_before": { ... },
  "scores_holdout_after":  { ... },
  "decision": "kept" | "rejected (no visible gain)" | "rejected (held-out regression)",
  "commit_sha": "abc1234" | null,
  "iteration_usd_cost": 1.23
}
```

### 4.3 CLI

```
python3 -m evals.auto_loop \
  --max-iterations 50 \
  --max-usd 20 \
  --max-hours 4 \
  --proposer claude-sonnet-4-5 \
  --judge opus \
  --resume <audit-dir>      # optional: continue from a previous run
```

Defaults: `--max-iterations 20 --max-usd 10 --max-hours 2 --proposer claude-sonnet-4-5 --judge opus`. Small default so a misconfigured first run doesn't burn the budget.

### 4.4 Slow-state discipline

The loop edits **only** these files (hardcoded allowlist):

- `plugin/skills/_shared/orchestrator-procedure.md`
- `plugin/skills/_shared/references/prompt-rubric.md`
- `plugin/skills/_shared/references/examples.md`

Off-limits:

- Preambles (per-skill inputs section — structural).
- Other references (`hook-patterns.md`, `tools-reference.md`, `settings-merge.md` — factual reference, not procedural).
- Anything under `evals/` (would let the loop optimize the metric instead of the skill).
- Anything outside `plugin/skills/_shared/`.

`sync_skills.py` runs after every applied edit. Pre-commit `--check` is the structural guarantee that nothing leaks into the generated trees.

### 4.5 Failure-target selection

`pick_target()` picks the lowest-scoring fixture in the visible-9, rotated over the bottom-3 to avoid the loop chasing one fixture forever:

```python
def pick_target(visible_baseline, recent_picks):
    bottom_3 = sorted(visible_baseline.entries, key=composite_score)[:3]
    return [f for f in bottom_3 if f.id not in recent_picks[-2:]][0]
```

Composite score: weighted sum of (code, install, fire) — same weights as a release decision. Restraint fixtures get max priority if they're in bottom-3 (since they're the largest headroom currently).

### 4.6 Cost accounting

Each iteration tracks per-API-call cost from `claude --print` exit metadata (or a coarse estimate: input_tokens × $X + output_tokens × $Y per model). USD cap is enforced *before* each iteration starts (lookahead estimate); if the next iteration would exceed the cap, the loop ends cleanly with a "cap reached" log entry.

## 5. Testing

- `evals/tests/test_auto_loop.py` — unit tests with the actual eval mocked:
    - `strictly_better` accepts a candidate that gains on one metric, no regression on others
    - `strictly_better` rejects a tie or regression beyond epsilon
    - `regresses` catches a held-out gating-metric drop
    - `pick_target` rotates over bottom-3 (no fixture picked twice in last 2)
    - `apply_edit` rejects an edit whose `anchor` isn't unique in the file
    - `audit.write_iteration` produces the expected JSONL row schema
- `evals/tests/test_edit_proposer.py`:
    - Proposer prompt assembly includes the right context blocks
    - Proposer response parsing handles all three operations (add/delete/replace)
    - Confidence < 4 short-circuits (skip apply, log "low confidence, declined to try")
- **End-to-end smoke test** (manual, not CI): a 3-iteration run with `--max-iterations 3 --max-usd 2` against the current main. Verifies the loop completes, audit log is structurally valid, git state is clean (committed kepts, no dirty rejects).

## 6. Risks & mitigations

| Risk | Mitigation |
|---|---|
| **Cost runaway** | Hard USD cap with lookahead estimate; abort before exceeding |
| **Catastrophic forgetting** | 1 edit / iteration (already practiced); git ratchet means each commit is independently validated; `git revert <sha>` restores any earlier state |
| **Overfit to visible-9** | Held-out gate; v0.5-β explicitly validates this before scaling iteration count |
| **Proposer loops on same bad edit** | Last-5 rejected edits passed into proposer prompt; novelty requirement |
| **Anchor collision / not found** | Reject the edit pre-application; log; don't burn an eval |
| **Verifier wall** (§8.6) | Loop *only* optimizes deterministic axes; `model` (LLM-judge) is visible to the proposer but not gating on ratchet |
| **Loop discovers an "unfair" win** (game the eval, not improve the skill) | Held-out gate catches some; periodic human review of kept commits before pushing to remote |
| **Slow-state allowlist bypass** | `apply_edit` rejects writes outside the allowlist; pre-commit `sync_skills --check` catches anything that escapes the allowlist check |
| **Sync script failure mid-iteration** | After edit + sync, run `sync_skills.py --check`; if it fails, treat as a rejected edit and reset |
| **Reproducibility** | Audit log records all model calls; re-running from the same dir + same models is directionally consistent (won't be bit-identical due to model nondeterminism) |

## 8. Exit criteria (the v0.5.0 tag)

- [ ] Loop completes 50+ iterations unattended without crashing
- [ ] At least one fixture's `code` score improves ≥ 1.5 points across the run
- [ ] Audit log JSON contains every iteration's edit, scores before/after, keep/reset decision, hypothesis
- [ ] Loop respects all three caps (verified by a deliberate cap-hit run)
- [ ] Reproducibility check: re-running from the same baseline produces keep-set overlap > 50%
- [ ] At least one auto-kept commit demonstrates a win the prompt-lab session didn't find (the loop *discovers* something — the project name pays off)

## 9. Open design questions (not blocking, want user input before plan)

| Q | Default | When to revisit |
|---|---|---|
| Proposer model: sonnet 4.5 vs opus? | sonnet (5× cheaper, v0.4 baseline shows clean proposals) | If sonnet plateaus at ≥10 iterations with no kept edit, escalate to opus for next batch |
| Strict-vs-soft holdout gate? | Strict (any negative delta on a gating metric rejects) | If too many candidates rejected on held-out noise rather than genuine regression, loosen epsilon |
| Allowlist (which files are editable)? | `orchestrator-procedure.md` + `prompt-rubric.md` + `examples.md` only | If the loop hits a ceiling, consider expanding to `hook-patterns.md` |
| Cost cap default? | $20 USD | After a few runs, tune to match the "useful run length" you actually want |
| `model` (LLM-judge) gating? | Not gating (advisory, visible to proposer) | Keep as advisory unless the deterministic axes saturate |

## 10. Connection to ROADMAP

This spec closes the v0.4 → v0.5 fork in the road. ROADMAP lists six v0.5 Path A components; we deliver five (skip the per-Step binary assertions per §3 non-goals) and inherit the substrate from v0.4 + v0.4.1 (eval, held-out, sync_skills, compactness). The v0.5.0 tag is "loop works end-to-end on this repo's skill"; v0.6+ generalizes the *target* (description optimization) and *runtime* (overnight, multi-skill).

If v0.5.0 ships clean, "self-improving-claude" stops being aspirational.
