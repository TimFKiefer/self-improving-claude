# v0.4.0 — Hook-Firing Behavioral Harness (Track 6.1) — Design Spec

- **Date:** 2026-05-27
- **Status:** Approved in brainstorm; awaiting user review before `writing-plans`.
- **Milestone:** v0.4.0 "Foundation for Autonomy", **Track 6.1** — the eval's "behavioral eyes."
- **Scope:** a new `evals/grade_behavior.py` that verifies a generated **command-hook** actually *fires* by driving its script with a synthetic stdin payload; per-fixture firing payloads on the command-hook fixtures; integration into the sandbox eval as a `fired` / `fire_rate` axis. No changes to `grade_code`/`grade_model`.
- **Companion:** `docs/superpowers/plans/2026-05-24-self-improving-claude-v0.4-backlog.md` §6.1; builds on SP-1 (`2026-05-26-v0.4.0-real-skill-sandbox-eval-design.md`).

---

## 1. Why this work exists

SP-1's sandbox eval stops at *"the proposal's script parses (`grade_code._check_script_parses`) and was written to disk (`install_rate`)."* A hook can pass every static check and still **no-op at runtime** — the v0.3.1/v0.3.2 "enforcement-shape" saga (a hook that passed every check yet failed to compel action) was precisely a *behavioral* failure a shape-eval is structurally blind to. `docs/knowledge/eval-methodology.md` §5 names this gap. Track 6.1 adds the behavioral layer: does the generated command-hook actually block/emit on the input it should catch — and **not** over-fire on a clean input?

Research (this session) established that of the 5 proposal forms, only the two **command-hook** forms (PreToolUse, PostToolUse) are *deterministically* verifiable without a live Claude session: they are standalone scripts that read a stdin JSON envelope and exit 0/2. `permissions.deny`/`permissions.ask` are settings rules (no script — evaluated inside Claude Code's permission engine), and `prompt-hook` is LLM-evaluated. Those need a live session and are out of scope here.

---

## 2. Goals

- **G1.** For a `command-hook` proposal, verify firing deterministically by running its script as a subprocess with a synthetic stdin payload — **no Claude session, no quota, no network**.
- **G2.** Test **both directions**: the *trigger* payload fires (exit 2; PostToolUse also emits stderr), and the *passthrough* (clean) payload does **not** over-fire (exit 0). This catches both no-op hooks and over-broad hooks.
- **G3.** Report firing as a new **orthogonal axis** — `fired: bool|None` per proposal and a `fire_rate` aggregate — **never folded into** the code or model grades (mirrors `install_rate`).
- **G4.** Additive: `grade_code`, `grade_model`, the sandbox pipeline, and the legacy path are unchanged.
- **G5.** Fully unit-tested with small real scripts in `tmp_path`; no `claude` invocation in the suite.

## 3. Non-goals

- **`permissions.deny`/`permissions.ask` behavioral verification** — no script to drive; the rule is evaluated by Claude Code's permission engine. Stays static-only (SP-1's `install_rate` already confirms the rule string landed in `settings.json`).
- **`prompt-hook` verification** — LLM-evaluated, needs a live session (or an approximating API call with different runtime). Reported `fired: None` ("unverified").
- **Live-session firing** (install the hook into a real Claude session and observe a real tool call get blocked) — the deeper 6.1 endpoint, deferred. This spec is the deterministic subprocess layer.
- **Changing `grade_code`/`grade_model`** or the legacy `run_one_entry` path.

## 4. Architecture & contracts

### 4.1 `evals/grade_behavior.py` (new)

```python
def run_firing_check(proposal: dict, firing_test: dict, *, timeout: float = 10) -> dict:
    """Drive a command-hook script with synthetic stdin payloads and verify firing.

    Returns:
      {
        "fired": bool,              # blocked_on_trigger AND passed_on_clean
        "blocked_on_trigger": bool, # script exited 2 on the trigger payload (+ stderr if required)
        "passed_on_clean": bool,    # script exited 0 on the passthrough payload
        "evidence": {"trigger_rc": int, "trigger_stderr": str, "clean_rc": int},
        "error": str | None,        # set when the script can't be run (lang/syntax/timeout)
      }
    Only meaningful for form == "command-hook"; callers pass fired=None for other forms.
    """
```

Behavior:
1. Write `proposal["script"]` to a temp file with the extension for `proposal["script_lang"]` (`python`→`.py`, `bash`→`.sh`, `javascript`→`.js`).
2. Run it as a subprocess (`python3 <f>` / `bash <f>` / `node <f>`) once per payload, feeding the payload JSON on **stdin**, `timeout`-bounded, capturing rc + stderr.
3. `blocked_on_trigger` = `trigger_rc == 2` and (if `firing_test["stderr_must_include"]` present) the substring is in stderr. `passed_on_clean` = `clean_rc == 0`. `fired` = both true.
4. On `FileNotFoundError` (interpreter missing), non-2/non-0 anomalies, or timeout → `error` set, `fired=False`.
5. Always delete the temp file. One run per payload, no retries.

### 4.2 `firing_test` fixture block (`dataset.json`, command-hook fixtures only)

```jsonc
"firing_test": {
  "trigger":     { "hook_event_name": "PreToolUse", "tool_name": "Bash",
                   "tool_input": { "command": "pnpm test" } },
  "passthrough": { "hook_event_name": "PreToolUse", "tool_name": "Bash",
                   "tool_input": { "command": "pnpm test:ci" } },
  "stderr_must_include": "test:ci"          // optional
}
```

The synthetic envelopes use the per-tool `tool_input` shapes from `docs/knowledge/tools-reference.md`. Added to the four command-hook fixtures:
- **001-pnpm-test-watcher** (PreToolUse Bash): trigger `pnpm test` → block; passthrough `pnpm test:ci` → allow.
- **005-format-on-write** (PostToolUse Write|Edit): trigger an Edit to a `.py` file → exit 2 + ruff-format stderr; passthrough an Edit to a `.md` file → exit 0.
- **006-rename-callers** (PostToolUse Edit|MultiEdit): trigger an Edit to a line defining an exported identifier → exit 2 + callers stderr; passthrough an unrelated Edit → exit 0.
- **008-secret-in-source** (PreToolUse Write|Edit): trigger a Write whose `content` contains `sk-...` → block; passthrough clean content → allow.

(PostToolUse hooks use exit 2 to feed stderr back; `blocked_on_trigger` therefore means "exit 2 + stderr" for both event types — the harness treats exit-2 uniformly as "fired.")

### 4.3 Integration (`evals/run.py` sandbox mode)

- `run_one_entry_sandbox`: for each proposal, if `proposal.get("form") == "command-hook"` **and** the entry has a `firing_test` → `fired = run_firing_check(proposal, entry["firing_test"])["fired"]`; else `fired = None`. Add `"fired": [...]` (aligned to proposals) to the per-entry result.
- `_aggregate_sandbox`: add `fire_rate` = (command-hook proposals that fired correctly) / (command-hook proposals where `firing_test` applied), or `None` if none applied. Reported alongside `install_rate`; **never folded into** `average_code`/`average_model`.
- `_main_sandbox`: print `fire_rate` in the summary line; include in the output JSON summary.

`grade_behavior` is a standalone module (like `grade_code`) — importable and testable independently of the sandbox runner.

## 5. Testing

`evals/tests/test_grade_behavior.py` — small real scripts written into `tmp_path`, run for real (these ARE the behavior under test; no mocking of the subprocess):
- A PreToolUse Python hook that `sys.exit(2)` when `tool_input.command` starts with `pnpm test` (but not `pnpm test:ci`) and `exit(0)` otherwise → `fired=True`, `blocked_on_trigger=True`, `passed_on_clean=True`.
- A no-op hook (always `exit(0)`) → `blocked_on_trigger=False`, `fired=False`.
- An over-broad hook (always `exit(2)`) → `passed_on_clean=False`, `fired=False`.
- `stderr_must_include` enforced: a hook that exits 2 but with the wrong stderr → `blocked_on_trigger=False`.
- A bash variant + a JS variant (skip the JS test if `node` is absent).
- A script that errors / wrong `script_lang` → `error` set, `fired=False`.

`evals/tests/test_run.py` (append): `run_one_entry_sandbox` sets `fired` for a command-hook entry with `firing_test` and `None` for a permissions form; `_aggregate_sandbox` computes `fire_rate` and keeps it separate from grades.

## 6. Risks & mitigations

| Risk | Mitigation |
|---|---|
| The firing check drives the **echoed** script (real-skill run under the forceful override), which may be malformed | `fired=False`/`error` correctly surfaces it — that's the signal, not a harness bug |
| `script_lang` mismatch (python script tagged bash) | run errors → `error` set, `fired=False` (correct) |
| Pathological/looping script | `timeout` (default 10s) → `error`, `fired=False` |
| Running untrusted generated scripts as subprocesses | they're model-generated hook bodies, already run in SP-1's sandbox under bypassPermissions; here they run with a synthetic stdin and a timeout, no args — low blast radius. (Could sandbox further later; not now.) |

## 7. Out of scope / deferred

- Behavioral verification of `permissions.*` and `prompt-hook` (needs a live session); the full live-session firing harness.
- Legacy-path (`run_one_entry`) integration — sandbox mode only.
