# v0.5.0 SP-α — Auto-Loop Smallest Viable Cut — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (inline) or superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax for tracking.
> **Cost discipline:** the smoke test runs single-fixture evals (`run.py --entry <id>`), not full-12. ~$1 total expected.

**Goal:** Implement the smallest viable cut of the auto-loop: edit-proposer + loop driver + ratchet + apply-edit + audit log. Single target fixture, no held-out gate, no rotation. Verified by a 5-iteration smoke test against fixture 010 using single-fixture evals.

**Architecture:** Four new modules under `evals/` (`auto_loop.py`, `edit_proposer.py`, `ratchet.py`, `audit.py`) plus tests. TDD for the deterministic components (ratchet, audit, apply_edit, prompt assembly). Manual smoke test for end-to-end verification. Reuses every v0.4 / v0.4.1 mechanism: `client_claude_cli.py` for proposer calls, `run.py --sandbox --entry <id>` for evals, `sync_skills.py` for slow-state regeneration, git for ratchet.

**Tech Stack:** Python 3.10+ stdlib (`subprocess`, `argparse`, `json`, `pathlib`, `signal`, `datetime`), `pytest`. Claude CLI (subscription OAuth) for the proposer.

**Spec:** `docs/superpowers/specs/2026-05-28-v0.5.0-auto-loop-design.md`.

## Task 1: `evals/ratchet.py` — pure score comparison (TDD)

- [ ] **Step 1.1:** Create `evals/ratchet.py` with two functions and an `EPSILON` constant table:

```python
EPSILON = {"average_code": 0.05, "install_rate": 0.0, "fire_rate": 0.05, "average_restraint": 0.0}
GATING_METRICS = list(EPSILON.keys())

def strictly_better(new: dict, old: dict) -> bool:
    """True iff at least one gating metric strictly up by > epsilon, none down by > epsilon."""

def regresses(new: dict, old: dict) -> bool:
    """True if any gating metric drops by > epsilon (held-out gate)."""
```

- [ ] **Step 1.2:** None-handling — if `old[m]` or `new[m]` is None, treat as "no change" on that metric (skip it). If new becomes None when old was a number, that's a regression.

- [ ] **Step 1.3:** Write `evals/tests/test_ratchet.py` covering:
  - one-metric-up-no-others-down → `strictly_better=True`
  - one-metric-up-one-other-down-beyond-epsilon → `strictly_better=False`
  - exact tie on all metrics → `strictly_better=False` (per SkillOpt §8.3)
  - within-epsilon "tie" on code → still false (not strictly better)
  - None handling — None→None is no change, None→number is improvement, number→None is regression
  - `regresses` returns True on any beyond-epsilon drop
  - `regresses` returns False on within-epsilon fluctuation

- [ ] **Step 1.4:** Run `pytest evals/tests/test_ratchet.py -v` — all green.

## Task 3: `apply_edit` — anchor-based file editor (TDD, in `auto_loop.py`)

- [ ] **Step 3.1:** Stub `evals/auto_loop.py` with imports + module-level `SLOW_STATE_ALLOWLIST` set:

```python
SLOW_STATE_ALLOWLIST = {
    "plugin/skills/_shared/orchestrator-procedure.md",
    "plugin/skills/_shared/references/prompt-rubric.md",
    "plugin/skills/_shared/references/examples.md",
}
```

- [ ] **Step 3.2:** Implement `apply_edit(edit: dict, repo_root: Path) -> tuple[bool, str]`:
  - Returns `(ok: bool, reason: str)` — `ok=True` on successful application, `ok=False` with a reason string on rejection.
  - Rejects if `edit["file"]` not in `SLOW_STATE_ALLOWLIST`.
  - Reads target file, counts occurrences of `edit["anchor"]` — rejects if not exactly 1.
  - Validates `edit["operation"] in {"add", "delete", "replace"}`.
  - For `add`: `edit["anchor_position"] in {"before", "after"}`; new content inserted accordingly.
  - For `replace`: anchor → new_content (1 occurrence).
  - For `delete`: anchor → "" (1 occurrence).
  - Bounds check: `new_content` ≤ 8 lines (per SkillOpt §8.1 edit-budget) — reject if exceeded.
  - On success, writes the file atomically (temp file + rename) and returns `(True, "applied")`.

- [ ] **Step 3.3:** Tests in `evals/tests/test_auto_loop.py`:
  - `apply_edit` rejects file outside allowlist (e.g. `evals/run.py`)
  - `apply_edit` rejects when anchor occurs zero times
  - `apply_edit` rejects when anchor occurs ≥ 2 times
  - `apply_edit` rejects when `new_content` > 8 lines
  - `apply_edit` succeeds for each of {add-before, add-after, replace, delete}; file content correct after each
  - File state preserved on rejection (no partial writes)

- [ ] **Step 3.4:** Run `pytest evals/tests/test_auto_loop.py::test_apply_edit -v`.

- [ ] **Step 5.1:** Implement `pick_target` (α version — fixed single fixture):

```python
def pick_target(visible_baseline: dict, target_fixture: str | None) -> dict:

```

- [ ] **Step 5.2:** Implement `run_iteration(state, audit, target_fixture, client, proposer_model, judge_model) -> tuple[bool, dict]`:

```
1. Snapshot current state of target fixture (baseline scores)
2. Build proposer input (fixture_failure block from baseline)
3. Call propose_edit(...)
   - If None (low confidence): write iteration record (decision=low_confidence), return
   - If ValueError parsing: write iteration record (decision=invalid_edit), return
4. apply_edit(...)
   - If rejected: write iteration record (decision=invalid_edit), return
5. Run sync_skills.py via subprocess
   - If fails: git reset, write iteration record (decision=sync_failed), return
6. Run single-fixture eval: run.py --sandbox --entry <target>
7. Score comparison: strictly_better(new, old)
   - If better: git add ; git commit ; record decision=kept, commit_sha
   - Else: git reset --hard HEAD ; record decision=rejected
```

- [ ] **Step 5.3:** SIGINT handler — catches Ctrl-C, writes final audit summary, exits cleanly. Use `signal.signal(signal.SIGINT, ...)`.

- [ ] **Step 5.4:** CLI via argparse:

## Risk-specific mitigations for α

| Risk | α mitigation |
|---|---|
| Proposer returns garbage | Strict JSON validation (Task 4.3) — reject pre-apply, log, count against iteration cap not budget |
| Anchor collision | `apply_edit` counts occurrences; reject if ≠ 1 (Task 3.2) |
| Sync failure mid-iteration | Run `sync_skills.py --check` after each apply; on failure, reset and log (Task 5.2 step 5) |
| Git dirty state at end | Every iteration ends with either `commit` or `reset --hard` (Task 5.2) |
| Cost overrun | α uses single-fixture evals (~$0.30 each × 5 × 2 = ~$3 max). No cap enforcement needed at this scale. |
| SIGINT during eval | Eval is a subprocess; SIGINT propagates; loop's handler writes summary and exits. Verified in Task 5.6. |
