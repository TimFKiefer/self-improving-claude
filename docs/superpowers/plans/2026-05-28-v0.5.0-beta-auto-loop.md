# v0.5.0 SP-β — Auto-Loop with Held-Out Gate + Rotation — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans (inline). Steps use checkbox (`- [ ]`) syntax for tracking.
> **Companion:** α plan at `docs/superpowers/plans/2026-05-28-v0.5.0-alpha-auto-loop.md`. β builds on α's foundation; does NOT replace any α module.

**Goal:** Lift v0.5.0-α from "works, but overfits" to "works, and the held-out gate catches overfit." Three structural additions: (1) held-out confirmation gate, (2) bottom-3 rotation over visible fixtures, (3) two small follow-ups from α's smoke tests. Plus a 20-iteration validation run that should exhibit at least one held-out-auto-rejection.

**Architecture:** No new modules. β is *additions* to existing α code:
- `evals/auto_loop.py:run_iteration` gains a held-out gate step between `strictly_better` and `git commit`.
- `evals/auto_loop.py:pick_target` learns to rotate over the bottom-3 lowest-scoring visible fixtures.
- `evals/auto_loop.py:run_iteration` adds a saturation pre-check to skip iterations on a maxed-out baseline.
- `evals/edit_proposer.py:parse_proposer_response` relaxes `anchor_position` to only apply when `operation == "add"`.
- New CLI flag `--rotate-bottom-n N` (default 3); `--target-fixture` becomes optional.

**Tech Stack:** Same as α — Python 3.10+ stdlib, `pytest`, Claude CLI (subscription OAuth).

**Spec:** `docs/superpowers/specs/2026-05-28-v0.5.0-auto-loop-design.md` (§4.1 held-out gate, §4.5 failure-target selection).

**Why:** Run-1 i=4 rejected on `anchor_position: null` for a non-`add` operation. The validator was over-strict — `anchor_position` is semantically meaningless for `delete` and `replace`.

```python
# Before (α):
if data["anchor_position"] not in ALLOWED_ANCHOR_POSITIONS:
    raise ValueError(f"invalid anchor_position: {data['anchor_position']}")

# After (β):

    if data.get("anchor_position") not in ALLOWED_ANCHOR_POSITIONS:
        raise ValueError(f"add requires anchor_position before|after, got: {data.get('anchor_position')!r}")
```

## Task 2: Saturation pre-check (α anomaly fix)

**Why:** Run-1 sonnet baseline on fixture 010 was already at code 10.0 / install 1.0 — no headroom. The proposer hallucinated a failure mode anyway. β should skip iterations when the target's baseline is at the ceiling for all gating metrics.

- [ ] **Step 2.1:** Define "saturated" in `evals/auto_loop.py`:

```python
def is_saturated(summary: dict) -> bool:
    """A target is saturated if all gating metrics are at their max (or unavailable)."""
    # code_max=10, install_rate=1.0, fire_rate=1.0 (or None), restraint=10 (or None)
    code = summary.get("average_code")
    install = summary.get("install_rate")
    fire = summary.get("fire_rate")
    restraint = summary.get("average_restraint")
    # None means "not applicable for this fixture"; only None or max counts as saturated
    return ((code is None or code >= 10.0 - 1e-6)
            and (install is None or install >= 1.0 - 1e-6)
            and (fire is None or fire >= 1.0 - 1e-6)
            and (restraint is None or restraint >= 10.0 - 1e-6))
```

- [ ] **Step 2.2:** In `run_iteration`, add a pre-check immediately after computing `baseline`:

```python
if is_saturated(baseline):
    record["decision"] = "skipped: saturated_baseline"
    record["edit"] = {}
    audit.write_iteration(record)
    return baseline, last_result
```

- [ ] **Step 2.3:** Tests in `test_auto_loop.py`:
  - `is_saturated` returns True for `{code:10, install:1, fire:None, restraint:None}`
  - `is_saturated` returns False for `{code:8.5, install:1, fire:None, restraint:None}`
  - `is_saturated` returns True for `{code:None, install:None, fire:None, restraint:10}` (restraint-only saturated)
  - `is_saturated` tolerates floating-point near-max (10.0 - 1e-9 still saturated)

- [ ] **Step 2.4:** Add `"skipped: saturated_baseline"` to the documented decision set in `evals/audit.py:REJECTION_PREFIX` (note: doesn't START with "rejected" — so it's not added to history fed back to the proposer; rotation will pick a different fixture next).

## Task 3: Bottom-3 rotation in `pick_target`

**Why:** Spec §4.5. Avoid the loop chasing one fixture forever. Picks the lowest-composite-score visible fixture not picked in the last 2 iterations.

- [ ] **Step 3.1:** Rewrite `pick_target` in `evals/auto_loop.py`:

```python
def pick_target(visible_baseline: dict, target_fixture: str | None,
                recent_picks: list[str], rotate_bottom_n: int = 3) -> str:
    """For β: if target_fixture is set, always return it. Otherwise rotate
    over the bottom-N lowest-composite-score visible fixtures, skipping any
    picked in the last 2 iterations to keep variety."""
    if target_fixture:
        return target_fixture
    entries = visible_baseline.get("entries", [])
    if not entries:
        raise ValueError("no visible entries to pick a target from")

    def composite(e):
        return (e.get("code_max", 0.0) or 0.0) + (e.get("install_rate", 0.0) or 0.0) * 10
    bottom = sorted(entries, key=composite)[:rotate_bottom_n]
    avoid = set(recent_picks[-2:])
    fresh = [e for e in bottom if e["id"] not in avoid]
    if not fresh:  # all bottom-N were picked recently — pick the oldest of them
        fresh = bottom
    return fresh[0]["id"]
```

- [ ] **Step 3.2:** Tests in `test_auto_loop.py`:
  - With 3 entries (compositions 3, 7, 9), `pick_target(..., recent_picks=[])` returns the lowest (composite 3)
  - With recent_picks containing the lowest, the next-lowest is picked
  - With ALL bottom-N picked recently, the oldest of bottom-N is picked (avoid empty result)
  - When `target_fixture` is set, always returned regardless of recent_picks (α behavior preserved)
  - `rotate_bottom_n` parameter respected

- [ ] **Step 3.3:** Update the existing α tests that called `pick_target(visible_baseline, target)` — add the `recent_picks=[]` argument. (Compatibility shim: default `recent_picks` to empty list so callers can omit.)

## Risk-specific mitigations for β

| Risk | β mitigation |
|---|---|
| Held-out eval cost ~doubles per kept iteration | Acceptable — only runs when visible already improved. Cap on `--max-iterations` keeps total cost bounded. |
| Rotation picks the same fixture twice if bottom-3 is small | `avoid` set caps at recent_picks[-2:] — works for any bottom-N ≥ 2. Tested. |
| Saturation pre-check wrongly skips a still-improvable target | Only triggers when ALL gating metrics are at max. Conservative. |
| Held-out baseline drifts during a long run | β does NOT re-baseline held-out — it's pinned at start. v0.5.0 may add periodic re-baselining if 50-iter shows drift. |
| 20-iter run hits API rate limits | Single bash command, no concurrent calls. If hit, restart with `--max-iterations N` from where we stopped (resume = v0.7). |
